## 核心结论

**LLM Wiki**（github.com/nashsu/llm_wiki，3300+ ⭐）是基于 Karpathy Gist 理念的跨平台桌面应用——把文档丢进去，LLM 自动生成结构化 Wiki 页面、知识图谱，并主动发现/补全知识缺口。比原版 Gist 多了图谱可视化、深度研究、Chrome 剪藏、向量检索四个核心能力。

**一句话定位**：传统 RAG 每次提问都从原始文档重检；LLM Wiki 先让 LLM 把文档"吃透"编译成 Wiki，后续直接对 Wiki 操作，知识复用率更高。

## 精华摘录

### 核心架构：两步链式思考录入

这是整个项目最核心的设计——比 Karpathy 原版的一步到位（边读边写）拆成了两步：

1. **分析阶段**：LLM 先通读文档，提取实体/概念/论点，发现与已有 Wiki 的关联和矛盾
2. **生成阶段**：拿着分析结果才写 Wiki——摘要页、实体页、概念页，更新索引，建立交叉引用

效果：一个来源录入 → 可能牵动 **10-15 个 Wiki 页面**自动更新。

**增量机制**：SHA256 哈希缓存，没改过的文件自动跳过；持久化队列崩了重启接着跑，失败重试 3 次。

### 知识图谱：四个关联维度

| 维度 | 权重 | 含义 |
|------|------|------|
| 直接链接 | x3.0 | 页面间 wikilinks 直接引用 |
| 来源重叠 | x4.0 | 两个页面引用同一原始文档 |
| Adamic-Adar | x1.5 | 有共同稀有关联节点 |
| 类型亲和度 | x1.0 | 同类型页面额外加分 |

技术栈：sigma.js + ForceAtlas2 布局 + Louvain 社区发现算法。

### 图谱洞察（最有价值的部分）

**意外关联**：跨社区/跨类型的意料之外连接——往往是认知突破的起点。

**知识缺口**：孤立页面、稀疏社区、桥接节点。每个缺口可一键触发深度研究——LLM 自动搜资料补全。

### 深度研究：知识库会自己补全自己

缺口被发现 → LLM 生成精准搜索词 → Tavily API 搜索 → LLM 综合分析写成研究页面 → 自动触发录入流程整合进知识网络。全程自动，搜索前有确认框可修改。

### Chrome 剪藏

- Readability.js 去广告/导航栏 → Turndown.js 转 Markdown → 自动触发录入
- 支持多项目选择
- 应用离线时也能预览提取内容

### 多阶段检索

分词搜索（CJK 二元组分词）→ 向量语义搜索（LanceDB ANN）→ 2跳关联遍历。召回率从 58.2% → 71.4%。

上下文窗口 4K-1M tokens，分配比：60% Wiki / 20% 聊天历史 / 5% 索引 / 15% 系统提示。

### 七个亮眼细节

1. **Purpose.md**：定义知识库的目标/范围/关键链接，LLM 每次录入和查询都参考它
2. **异步审核**：拿不准的内容进入待审核列表，不直接主观
3. **清理与摘要**：自动维护 Wiki 摘要页，来源/索引参数/错误全部清晰
4. **多格式支持**：PDF（Rust解析）、DOCX、PPTX、Excel、图片直接索引入
5. **完全兼容 Obsidian**：生成标准 Obsidian Vault，含 `.obsidian/` 配置
6. **完全离线可用**：本地文档全部离线处理
7. **不需要配置命令行/Agent**：普通用户可直接上手

## Deep Compression

**一句话总结**：LLM Wiki = Karpathy Gist 的桌面产品级实现，用两步链式录入 + 四维知识图谱 + 自动缺口补全，让个人知识库从"文档仓库"升级为"会自己生长的 Wiki"。

**一个最有价值的点**：图谱洞察——不是被动检索，而是主动发现跨领域的意外关联和知识缺口，并自动补全。这把知识管理从"我找知识"变成了"知识找我"。

**一个警示**：依赖 Tavily API 和云端大模型，完全离线场景受限。

## 与 Hermes wiki 工作流的关联

当前 Hermes 用 Obsidian + 手工维护知识库。LLM Wiki 的设计理念可以借鉴：

- **两步链式** → Hermes 的 raw → processed 双阶段归档流程已在做，但自动化程度可提升
- **Purpose.md** → 可在 wiki 根目录加 `purpose.md` 定义当前项目的知识边界
- **图谱洞察** → Hermes 目前没有自动发现知识缺口的能力，可作为未来技能方向
- **Obsidian 兼容** → 与现有 wiki 完全兼容，两个工具可协同

**结论**：可作为 Hermes 知识管理工作流的未来演进方向参考。

# 一个人如何写出 60 万行代码？YC 总裁开源了他的"虚拟工程团队"

> "我可能从去年 12 月到现在，连一行代码都没敲过。"
> 
> —— Andrej Karpathy，2026 年 3 月

讲真，听到 Karpathy 这句话时，我整个人都愣住了。

**一个人，到底怎么能像二十个人的团队那样交付？**

直到我发现了 **gstack** —— 这个由 Y Combinator 总裁 Garry Tan 亲自打造的开源项目。

说实话，看完它的 README，我只想说：**工具的革命，真的来了。**

* * *

## 01 同一个人，不同的时代

  

Garry 在 README 里放了两张对比图，特别有意思。

**2013 年**，他在 YC 构建 Bookface 时，全年 772 次贡献。

**2026 年**，使用 gstack 后，至今 1237 次贡献。

用他的话说：

> "同一个人。不同的时代。**区别在于工具。**"

过去 60 天，他用 gstack 完成了什么？

*   **60 万 + 行**生产代码（35% 是测试）
    
*   **每天 10,000-20,000 行**
    
*   **兼职完成**（同时全职运营 YC）
    

这不是炫技。这是 AI 时代单人开发的真实生产力。

## 02 gstack 是什么？

  

这么说吧，gstack 不是一堆工具的集合。

它是一套**完整的软件工程流程**。

核心主张就一句话：

> **一个配备正确工具的独立开发者，可以比传统团队移动更快。**

gstack 把 Claude Code 变成了一支虚拟工程团队，包含这些角色：

**• CEO/产品顾问** — `/office-hours`, `/plan-ceo-review` — 重新定义问题，找到更好的产品方向

**• 工程经理** — `/plan-eng-review` — 锁定架构，评审代码质量、测试、性能

**• 高级设计师** — `/plan-design-review`, `/design-review` — 发现 UI 不一致，生成交互式评审

**• QA 负责人** — `/qa` — 打开真实浏览器，系统性测试并修复

**• 首席安全官** — `/cso` — OWASP Top 10 + STRIDE 威胁建模

**• 发布工程师** — `/review`, `/ship`, `/land-and-deploy` — PR 评审、自动发布、部署验证

**• 调试专家** — `/investigate` — 系统性根因分析

**• 文档工程师** — `/document-release` — 自动更新所有文档

**23 个专业角色，8 个效率工具，全部是 slash 命令，全部 Markdown，全部免费，MIT 许可。**

## 03 核心工作流：像 Sprint 一样运行

  

gstack 的精髓，真不在于工具数量。

在于**流程设计**。

```
Think → Plan → Build → Review → Test → Ship → Reflect
```

每个环节都有对应的技能，而且**环环相扣**：

*   `/office-hours` 写出的设计文档，会被 `/plan-ceo-review` 读取
    
*   `/plan-eng-review` 写出的测试计划，会被 `/qa` 执行
    
*   `/review` 发现的 bug，会被 `/ship` 验证修复
    
*   `/document-release` 确保文档永远与代码同步
    

用 Garry 的话说：

> **没有环节会被遗漏，因为每一步都知道前一步做了什么。**

## 04 真实案例：从"日程简报 App"到"个人首席 AI 官"

  

Garry 在文档里记录了一个案例，我特别喜欢。

**用户说**："我想做一个日程简报 App。"

换作平时的 AI，可能直接就开始写代码了。

但运行 `/office-hours` 后，AI 没有这么做。它问了 6 个"逼问式"问题：

*   具体哪些场景让你痛苦？（不是假设的痛点）
    
*   你现在是怎么解决的？
    
*   什么情况下现有方案会失效？
    

**用户回答**："多个 Google 日历，活动信息过期，地点错误。每次准备会议都要花很久，结果还不靠谱。"

然后 AI 的回应，让我印象深刻：

> "我要挑战一下你的 framing。你说要'日程简报 App'，但你实际描述的是一个**个人首席 AI 官**。"

接下来 AI 做了四件事：

1.  提取出用户没意识到的 5 个能力需求
    
2.  挑战了 4 个前提假设
    
3.  给出 3 种实现方案，附带工作量评估
    
4.  **建议**："明天就发布最窄的 MVP，从真正可用的日程简报开始。完整愿景是 3 个月的项目。"
    

**8 个命令，端到端完成。**

Garry 说得好：

> "这不是 copilot，这是一个团队。"

## 05 几个让我印象深刻的技能

  

### `/design-shotgun`：视觉探索

这个功能特别酷。

你跟它说"给我看选项"，它会生成 4-6 个 AI 设计变体，在浏览器中并排展示。

你可以选择喜欢的，留下反馈：

*   "更多留白"
    
*   "标题更大胆"
    
*   "去掉渐变"
    

然后它会生成新一轮。

最妙的是 **Taste Memory（品味记忆）** —— 几轮之后，它开始偏向你真正喜欢的风格。

不再需要用文字描述愿景然后祈祷 AI 理解。你看到选项，选择好的，视觉迭代。

![](/assets/img/news/-4.jpg)

### `/qa`：真正的浏览器测试

这是 Garry 说的"巨大解锁"。

AI 打开真实的 Chromium 浏览器，点击你的应用，找到 bug，修复它们，生成回归测试，然后验证修复。

**"I SEE THE ISSUE" —— 然后真的修复了。**

就因为这个，他可以从 6 个并行 worker 提升到 12 个。

### `/codex`：跨模型第二意见

让 OpenAI 的 Codex CLI 对你的代码进行独立评审 —— 完全不同的 AI 看同一个 diff。

三种模式：

*   **Code Review**：pass/fail 门禁
    
*   **Adversarial Challenge**：主动尝试破坏你的代码
    
*   **Open Consultation**：开放式咨询
    

当 `/review`（Claude）和 `/codex`（OpenAI）都评审过同一个分支后，你会得到**跨模型分析**，显示哪些发现是重叠的，哪些是每个模型独有的。

### `/document-release`：你从未拥有过的文档工程师

这个功能，说实话，是我一直想要的。

它读取项目中每个文档文件，交叉引用 diff，自动更新所有过时的内容。

README、ARCHITECTURE、CONTRIBUTING、CLAUDE.md、TODOS —— **全部自动保持最新**。

现在 `/ship` 会自动调用它，文档不再需要额外命令就能保持同步。

## 06 并行 Sprint：10-15 个同时运行

  

gstack 在一个 Sprint 中很强大。

**在十个同时运行时，是变革性的。**

Garry 使用 Conductor 并行运行多个 Claude Code 会话：

*   一个会话在 `/office-hours` 新想法
    
*   一个在 `/review` PR
    
*   一个在实现功能
    
*   一个在 `/qa` 测试环境
    
*   另外六个在其他分支
    

**他通常同时运行 10-15 个并行 Sprint。**

Sprint 结构是让并行工作的关键。没有流程，十个 agent 是十个混乱源。有了流程 —— Think, Plan, Build, Review, Test, Ship —— 每个 agent 都知道该做什么、何时停止。

**你像 CEO 管理团队一样管理它们：只关注重要的决策，其余让它们自己运行。**

## 07 安全与边界

  

gstack 不是无脑执行。它有严格的安全机制。

*   **`/careful`**：在执行破坏性命令前警告（rm -rf, DROP TABLE, force-push）
    
*   **`/freeze`**：调试时将编辑锁定在一个目录内
    
*   **`/guard`**：同时激活上述两者
    
*   **`/investigate`**：自动冻结到正在调查的模块
    

还有一条铁律：

> **没有调查就没有修复。**

`/investigate` 会追踪数据流、测试假设，在 3 次修复失败后停止。

## 08 安装与使用

  

gstack 支持 8 个 AI 编程 agent，不仅限于 Claude Code：

*   **Claude Code** → `~/.claude/skills/gstack/`
    
*   **OpenAI Codex CLI** → `~/.codex/skills/gstack*/`
    
*   **OpenCode** → `~/.config/opencode/skills/gstack*/`
    
*   **Cursor** → `~/.cursor/skills/gstack*/`
    
*   **Factory Droid** → `~/.factory/skills/gstack*/`
    
*   **Slate** → `~/.slate/skills/gstack*/`
    
*   **Kiro** → `~/.kiro/skills/gstack*/`
    

**30 秒安装**（以 Claude Code 为例）：

```
git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
cd ~/.claude/skills/gstack && ./setup
```

然后在 CLAUDE.md 中添加 gstack 部分，指定使用 `/browse` 技能进行所有网页浏览。

**团队使用**：

```
cd ~/.claude/skills/gstack && ./setup --team
# 然后在你的 repo 中
~/.claude/skills/gstack/bin/gstack-team-init required
git add .claude/ CLAUDE.md && git commit -m "require gstack for AI-assisted work"
```

没有 vendored 文件，没有版本漂移，没有手动升级。每次 Claude Code 会话开始时都会快速自动检查更新（每小时限流一次，网络失败安全，完全静默）。

## 09 设计哲学

  

gstack 背后有几条核心设计原则，我觉得值得每个开发者思考。

**Boil the Lake（煮干湖水）**

不要试图一次性理解整个代码库。聚焦于当前任务相关的最小上下文。

**Search Before Building（先搜索再构建）**

在写任何代码之前，先搜索现有实现、文档、类似问题。

**三层知识**

1.  **瞬时知识**：当前会话中的上下文
    
2.  **项目知识**：`/learn` 技能管理的项目特定模式
    
3.  **全局知识**：通过 `/retro` 积累的工程经验

## 10 为什么开源？

  

"我开源了我构建软件的方式。你可以 fork 它，让它成为你的。"

Garry 在 README 最后写了这样一句话：

> "如果你想批评免费的开源软件 —— 欢迎你，但我更希望你先试试。"

gstack 完全免费，MIT 许可，没有高级版，没有等待名单。

**它包含可选的使用遥测**（默认关闭）：

*   只收集：技能名称、持续时间、成功/失败、gstack 版本、操作系统
    
*   绝不收集：代码、文件路径、仓库名称、分支名称、提示词、任何用户生成内容
    

数据存储在 Supabase（开源 Firebase 替代方案），schema 公开可查。

## 11 一些思考

  

写到这里，我想分享几点个人思考。

**AI 时代的开发者角色变化**

gstack 展示了一个趋势：**开发者正在从"写代码的人"变成"定义问题的人"。**

当 AI 可以写代码、测试、审查、部署时，人类的价值在于：

*   定义正确的问题
    
*   做出品味决策
    
*   判断何时足够好
    
*   在关键时刻介入
    

**流程比工具更重要**

很多 AI 工具是"功能集合"，但 gstack 是"流程封装"。

它把软件工程的 best practice 编码进了工具链：

*   先思考再编码
    
*   设计文档驱动开发
    
*   测试先行
    
*   文档同步
    
*   事后回顾
    

用一句话总结：

> **工具会过时，流程永存。**

**单人团队的可能性**

gstack 证明了一件事：

> **一个人 + 正确的工具 = 一个团队。**

但这不意味着传统团队会消失。它意味着：

*   小团队可以做更大的事
    
*   创始人可以保持动手能力
    
*   快速原型和验证变得极其便宜
    
*   创新门槛大幅降低
